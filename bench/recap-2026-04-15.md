## FSM journey

Five phases, each bench-gated on every fixture:

```
Phase 1   21de4d6   extract pure helpers → qa_agent/runtime/{evidence,
                   loop_detect, messages, mm_popup, tx_trigger}.py
Phase 2   e716cf0   AgentCtx dataclass + on_* proxies
Phase 3a  d94a511   generic FSM dispatcher + AgentState/Event enums
Phase 3b  9a83ce8 + ddc1e3e + ea77639
                   extract snapshot_page, evidence_verdict, loop_check,
                   vision_retry actions (still called from for-loop)
Phase 3c  b0453e3   AGENT_TRANSITIONS + fsm_actions.py; run_task body
                   replaced with `fsm.send(START)` + queue drain
```

`agent.py` went from **651 → 199 lines**. Net new code in
`qa_agent/runtime/` (fsm, states, transitions, fsm_actions, ctx,
actions, evidence, loop_detect, mm_popup, tx_trigger, messages):
**~1 550 lines**. The code grew overall — each piece now has a
docstring + a clear boundary — but the run-time graph is the same.

### What the FSM buys

- **No more hidden transitions**: every path through the agent is a
  row in `AGENT_TRANSITIONS`. Grep for an event, see every handler.
- **Dead-branch test is trivial**: BFS from IDLE through the table →
  set of reachable states. If DONE_PASS, DONE_FAIL, ERROR are all
  reachable, coverage is complete. (Unit test pending.)
- **`on_step` instrumentation is uniform**: the dispatcher has one
  emit path (`_emit_step(ctx)` called by step-boundary actions),
  instead of ten scattered assignments around `continue`/`break`.
- **Each action is pure-ish**: takes only `ctx`, emits one event.
  Trivial to unit-test in isolation (no Playwright needed for
  evidence_verdict / loop_check).

### What it doesn't fix

- Prompt quality — orthogonal. The v2a anti-pattern-rules merge
  already landed the only prompt win we identified.
- Token cost — FSM routing is strictly cleaner, but LLM calls per
  step are the same.
- Flake — FSM doesn't hallucinate, but WB can still rotate products.
